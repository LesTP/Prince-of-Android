# State Trace Binary Format

## Overview

State trace files (`.trace`) are binary dumps of SDLPoP game state, captured frame-by-frame during replay execution. Each frame captures the complete game simulation state needed to verify behavioral equivalence between the C original and the Kotlin port.

Generated by: `dump_frame_state()` in `SDLPoP/src/seg000.c` (when compiled with `-DDUMP_FRAME_STATE`)

## Frame Structure

Each frame is **310 bytes** with the following layout:

| Offset | Size | Type | Field | Description |
|--------|------|------|-------|-------------|
| 0 | 4 | dword | frame_number | Sequential frame counter (0-indexed) |
| 4 | 16 | char_type | Kid | Player character struct |
| 20 | 16 | char_type | Guard | Active guard character struct |
| 36 | 16 | char_type | Char | Current character being processed |
| 52 | 2 | word | current_level | Level number (1-14) |
| 54 | 2 | word | drawn_room | Currently drawn room |
| 56 | 2 | short | rem_min | Timer: remaining minutes |
| 58 | 2 | word | rem_tick | Timer: remaining ticks (720 ticks/minute) |
| 60 | 2 | word | hitp_curr | Kid's current hit points |
| 62 | 2 | word | hitp_max | Kid's maximum hit points |
| 64 | 2 | word | guardhp_curr | Guard's current hit points |
| 66 | 2 | word | guardhp_max | Guard's maximum hit points |
| 68 | 30 | byte[30] | curr_room_tiles | Current room tile types (3 rows × 10 cols) |
| 98 | 30 | byte[30] | curr_room_modif | Current room tile modifiers |
| 128 | 2 | short | trobs_count | Number of active animated tiles (trobs) |
| 130 | 90 | trob_type[30] | trobs | Animated tiles array (3 bytes × 30) |
| 220 | 2 | short | mobs_count | Number of active movable objects (falling tiles) |
| 222 | 84 | mob_type[14] | mobs | Movable objects array (6 bytes × 14) |
| 306 | 4 | dword | random_seed | RNG state |
| **Total** | **310** | | | |

## Nested Type Structures

### char_type (16 bytes)

Character state struct. Fields:

| Offset | Size | Type | Field | Description |
|--------|------|------|-------|-------------|
| 0 | 1 | byte | frame | Animation frame ID |
| 1 | 1 | byte | x | X position (fine-grained) |
| 2 | 1 | byte | y | Y position (fine-grained) |
| 3 | 1 | sbyte | direction | Facing direction (0=right, -1=left) |
| 4 | 1 | sbyte | curr_col | Current tile column (0-9) |
| 5 | 1 | sbyte | curr_row | Current tile row (0-2) |
| 6 | 1 | byte | action | Current action (0=stand, 1=run_jump, etc.) |
| 7 | 1 | sbyte | fall_x | Horizontal falling speed |
| 8 | 1 | sbyte | fall_y | Vertical falling speed |
| 9 | 1 | byte | room | Current room number |
| 10 | 1 | byte | repeat | Animation repeat counter |
| 11 | 1 | byte | charid | Character ID (0=kid, 2=guard, etc.) |
| 12 | 1 | byte | sword | Sword status (0=sheathed, 2=drawn) |
| 13 | 1 | sbyte | alive | Alive status (1=alive, 0=dead, <0=dying) |
| 14 | 2 | word | curr_seq | Current animation sequence ID |
| **Total** | **16** | | | |

### trob_type (3 bytes)

Animated tile state (chompers, spikes, gates, etc.):

| Offset | Size | Type | Field |
|--------|------|------|-------|
| 0 | 1 | byte | tilepos |
| 1 | 1 | byte | room |
| 2 | 1 | sbyte | type |

### mob_type (6 bytes)

Movable object state (falling floors, etc.):

| Offset | Size | Type | Field |
|--------|------|------|-------|
| 0 | 1 | byte | xh |
| 1 | 1 | byte | y |
| 2 | 1 | byte | room |
| 3 | 1 | sbyte | speed |
| 4 | 1 | byte | type |
| 5 | 1 | byte | row |

## Data Types

C type sizes (consistent across x86 and ARM64 via SDL typedefs):

- `byte` = Uint8 (1 byte, unsigned)
- `sbyte` = Sint8 (1 byte, signed)
- `word` = Uint16 (2 bytes, little-endian)
- `dword` = Uint32 (4 bytes, little-endian)

## Byte Ordering

All multi-byte values are **little-endian** (LSB first), per SDL's `SDL_BYTEORDER == SDL_LIL_ENDIAN` requirement.

## File Size Calculation

Total file size = `310 bytes/frame × N frames`

Example: `basic_movement.trace` = 123,070 bytes = 310 × 397 frames

## Determinism Verification

To verify replay determinism:
1. Run the same replay twice with the same `seed=N`
2. Binary compare the two trace files
3. If identical byte-for-byte → deterministic ✅
4. If divergent → identify first diverging byte offset, calculate frame number and field

## Usage in Test Oracle

The Kotlin port's replay runner will:
1. Parse a `.p1r` replay file
2. Feed recorded inputs into ported game logic frame-by-frame
3. Dump equivalent state trace using same binary format
4. Compare Kotlin trace against C reference trace byte-by-byte
5. On divergence: report frame number, field name, expected vs actual values

This is the core autonomous porting pipeline validation mechanism.
