# Step 16e — Level Screenshot Visual Comparison Status

## Overview
We are comparing Kotlin-rendered level screenshots against C (SDLPoP) reference screenshots
for all 14 levels of Prince of Persia. The goal is to identify and fix rendering differences.

## Completed Fixes

### 1. NO_TRANSP Alpha Fix (done in prior session)
Wall decoration sprites had light rectangular backgrounds because `Blitters.NO_TRANSP`
wasn't setting the alpha channel. Fixed in `SpriteRenderer.blitPixel()`.

### 2. Room Height Cropping to 189px (done in prior session)
Dark stripe at bottom of rooms. Fixed by copying only `ROOM_STRIDE_Y` (189) rows
instead of the full 200-pixel render height. The bottom 11 pixels are the status bar area.

### 3. Palace Wall Colors (FIXED in this session)
**Problem**: Palace levels (4+) showed rainbow-colored brick walls instead of yellow/beige.

**Root cause**: `SpriteRenderer.paletteColor()` masked palette indices with `0x0F`,
reducing the 256-entry VGA palette to 16 entries. Wall color indices like `0x61`
(generated by `genPalaceWallColors()`) became `0x01` (dark blue) instead of the correct
yellow shade from palette row 6 (0x60-0x6F).

**Fix**:
- Expanded `SpriteRenderer` palette from 16 to 256 entries
- Changed `paletteColor()` mask from `0x0F` to `0xFF`
- Added `buildFullPalette()` in `LevelScreenshotGenerator` that populates all 256 entries
  from loaded chtab palettes using each catalog's `DatPalette.rowBits` to determine which
  palette rows it occupies
- Added `level_var_palettes` (resource 20 from PRINCE.DAT) override for levels with
  `tblLevelColor != 0`, matching C's `set_pal_arr(0x50/0x60, ...)` logic
- Added C's `init_game_main()` palette overrides: VGA entry 6 → tan mortar color
  `(0x30, 0x26, 0x14)` and entry 12 → blood/hurt color `(0x38, 0x00, 0x0C)`
- Fixed sprite index-0 color preservation in `AssetImages.paletteColorToArgb()`:
  stores actual RGB with alpha=0, so NO_TRANSP blit shows the correct color
- Fixed render table draw order in `RenderTableFlusher.flushTables()` to match C:
  wipes(1) drawn before midtable (FIX_BLACK_RECT behavior)
- Verified: Level 4 (palace, tblLevelColor=0) and Level 8 (dungeon, tblLevelColor=2)
  both match C reference

### 4. Guard Sequence Fix (partial — frame resolution works, rendering doesn't)
**Problem**: `guardsSeqHi = 0xFF` in level data caused `Char.currSeq = 255 + (255<<8) = 65535`.
`seqtblRead(65535)` returned 0 (out of bounds), producing `Char.frame = 0` →
`getFrameInternal` with negative index → `image = 255` (invisible).

**Fix applied**: Zero out `guardsSeqHi` when 0xFF before calling `checkShadow()`.
Result: Frame correctly resolves to 166 → curFrame.image = 17 (valid guard sprite).

### 5. Guard Direction Sign Extension (applied but insufficient alone)
**Problem**: `guardsDir` loaded as unsigned byte (255) into `Char.direction` (Int).
In C, `Char.direction` is `sbyte` so 0xFF = -1. The comparison `Char.direction < Dir.RIGHT`
gives different results: C: `-1 < 0` = true; Kotlin: `255 < 0` = false.

**Fix applied**: Changed `parseLevel()` to read `guardsDir` with `s8()` (sign-extending).
But this alone doesn't fix guard rendering (see below).

## Outstanding Issues

### A. ALL Dynamic Objects Missing (guards, kid, torch flames, potion steam, level doors)

**This is the BIG remaining issue.**

#### Architecture of C's `redraw_screen(1)`:
```
redraw_screen(1):
  Pass 1:  redraw_room() + draw_tables()     ← STATIC tiles (we have this)
           clear_tables()
  Pass 2:  draw_moving() + draw_tables()      ← DYNAMIC objects (WE ARE MISSING THIS)
```

#### What `draw_moving()` does:
```
draw_moving():
  draw_mobs()           → torch flames, chomper animation, potion bubbles/steam, level doors
  draw_people():
    draw_kid()          → places kid sprite
    draw_guard()        → places guard sprite (we call this, but it's not enough alone)
  redraw_needed_tiles() → iterates all 30 tiles, for marked tiles calls
                          draw_objtable_items_at_tile(tilepos) which transfers
                          objtable entries → midtable for actual rendering
```

#### What we currently call:
```kotlin
// In renderRoom():
Seg008.drawGuard()   // adds guard to objtable ✓
Seg008.drawRoom()    // does static tile pass (Pass 1) ✓
// MISSING: draw_mobs(), draw_kid(), redraw_needed_tiles()
```

#### Why guards don't appear even though they're in objtable:
The guard IS added to objtable with correct sprite (chtab=5, id=17). BUT:
- Its `tilepos` = 30 (invalid, must be 0-29 for a visible tile)
- `markObjTileRedraw()` only marks tiles 0-29 for redraw
- During `drawRoom()`'s tile iteration, no tile matches tilepos=30
- Therefore `drawObjtableItemsAtTile()` never picks up the guard
- Guard never reaches midtable → never rendered

#### Guard Position Mystery (tilepos=30):
- `guardsX = 255` for ALL 24 rooms in level data (confirmed identical in both
  SDLPoP and app LEVELS.DAT — same file, byte-for-byte identical)
- C also reads `guardsX=255` yet C screenshots show guards at correct positions
- With `Char.x = 255`: `charDxForward(-1) = 1 + 255 = 256` → `objX = 396` → off-screen
- Even in C, x=255 should produce off-screen coordinates, yet guards appear
- Possible explanations:
  1. C's `redraw_needed_tiles()` has additional logic beyond what `drawRoom()` does
  2. C's byte-width `Char.x` (unsigned char) causes wrapping that Kotlin's Int doesn't
  3. `determineCol()` in `addGuardToObjtable()` computes differently in C due to
     signed/unsigned intermediate calculations
  4. Something in `draw_mobs()` or the second `draw_tables()` pass handles this

### B. Torch Animation
Torches show static base frames. In C, `draw_mobs()` adds animated torch flame sprites.
This is part of the `draw_moving()` pass we're missing.

### C. Potion Steam/Bubbles
We set `curr_room_modif` bit 0 for potions (matching C's `switch_to_room()`), but the
actual bubble rendering happens in `draw_mobs()` which we don't call.

### D. Kid Not Shown
C's `switch_to_room()` doesn't explicitly set up the kid for screenshot rendering.
The kid appears because the game state already has the kid positioned. For screenshots,
we'd need to set up Kid position from `level.startPos`/`level.startDir`.

### Guard Fields in LEVELS.DAT (level 1):
```
guardsTile: [30,30,17,30,30,30,30,30,30,30,30,30,30,30,30,30,30,30,30,30,6,30,30,30]
guardsDir:  [FF,00,FF,FF,00,00,FF,00,00,00,00,FF,00,00,00,FF,00,00,FF,00,FF,FF,00,00]
guardsX:    [FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF]
guardsSeqLo:[FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF]
guardsSkill:[FF,FF,00,01,FF,FF,FF,FF,00,01,FF,FF,01,01,FF,FF,01,FF,FF,01,00,FF,FF,FF]
guardsSeqHi:[FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF,FF]
```
- guardsTile=30 means "no guard in this room" (tilepos 30 is invalid)
- Room 3 (index 2): guardTile=17 (row 1, col 7), guardsDir=0xFF (LEFT), guardsX=0xFF
- Room 21 (index 20): guardTile=6 (row 0, col 6), guardsDir=0xFF (LEFT), guardsX=0xFF
- ALL guardsX/guardsSeqLo/guardsSeqHi are 0xFF — these are uninitialized defaults

### Direction Convention:
- C: `dir_0_right = 0`, `dir_FF_left = -1` (sbyte), `dir_56_none = 0x56`
- Kotlin: `Directions.RIGHT = 0x00`, `Directions.LEFT = -1`, `Directions.NONE = 0x56`
- Level data stores 0xFF (unsigned). C's sbyte assignment converts to -1 automatically.
  Kotlin needs explicit sign extension (now done via `s8()` in `parseLevel()`).

### Sequence Table:
- `SEQTBL_BASE = 0x196E`
- `seqtblOffsets[77]` (seq_77_guard_stand_inactive) = `SEQTBL_BASE + 58` = 6568
- `seqtbl[58] = 166` (frame_166_stand_inactive)
- `playSeq()` reads seqtbl bytecodes. Non-opcode values are frame numbers.

### Frame Table:
- Guard frames: `frameTblGuard[0..40]`, accessed as `frame - 149`
- Frame 166 → index 17 → `FrameType(image=17, sword=0xC9, dx=-1, dy=1, flags=0x48)`
- Out-of-bounds index → `FrameType(image=255, ...)` (invisible sentinel)
